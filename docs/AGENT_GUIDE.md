# AGENT_GUIDE

이 문서는 이 리포에서 작업하는 모든 AI 에이전트의 행동 규칙이다.
세션을 시작하면 이 문서와 `docs/PROGRESS.md`를 먼저 읽는다.

## 골든 루프 (모든 작업의 기본 단위)

1. **READ** — `docs/PROGRESS.md`를 읽고 Current Phase와 Status를 확인한다.
   - Status가 `AWAITING REVIEW`면: 개발자 검토 대기 중. 작업하지 말고 알린다.
   - Status가 `DONE`이면(프로젝트 전체): 할 일 없음.
2. **LOAD** — 현재 phase 파일(`docs/phases/NN-*.md`) 하나만 읽는다.
   필요 시 SPEC.md / ARCHITECTURE.md / CONVENTIONS.md를 참조한다.
3. **PLAN** — 그 phase의 `완료 조건`을 만족시킬 계획을 세우고 PROGRESS.md를
   갱신한다: Status를 `IN PROGRESS`로, Current Task를 착수할 작업 ID(첫 미완
   작업)로, Working Notes에 계획 1~2줄.
4. **BUILD** — phase의 `변경 범위` 안에서만 구현한다. SPEC.md의 제외 범위를
   지킨다. 작업(PNN-TNN) 하나가 끝날 때마다 phase 파일의 체크박스를 체크하고
   Current Task를 다음 작업으로 갱신한다.
5. **SELF-CHECK** — `완료 조건`을 항목별로 스스로 점검한다.
   충족 못 한 항목은 Blockers에 적고 정지한다 (Status는 `IN PROGRESS` 유지).
   이전 세션의 Blockers가 해소됐으면 `none`으로 되돌린다.
   Blockers에 "개발자 선택/제공 대기"가 있으면 **에이전트가 대신 정하지
   않는다** — 상황이 바뀌어 선택지가 생겼어도(예: 접근 정보가 새로 제공됨)
   선택지를 먼저 제시하고 확인받은 뒤 진행한다.
6. **UPDATE** — PROGRESS.md를 갱신한다:
   Status를 `AWAITING REVIEW`로, Completed 체크(= 구현 완료 표시 — 검토 승인
   아님), Current Task를 `none (검토 대기)`으로, Working Notes에 요약(한 것 /
   만든 파일 / 다음 세션 참고사항), phase 파일의 `개발자 테스트` 항목을
   Developer Test 섹션에 복사.
7. **COMMIT & STOP** — 변경을 PROGRESS.md까지 포함해 CONVENTIONS.md 형식으로
   커밋하고 정지한다. Current Phase를 스스로 올리지 않는다.
   다음 phase는 개발자가 검토 후 넘긴다.

**한 번에 정확히 한 phase.** 절대 다음 phase를 이어서 진행하지 않는다.
작업(task)의 상세는 별도 문서가 아니라 phase 파일의 `작업` 체크리스트가 겸한다.

## 세션 시작 메뉴

사용자가 구체적 지시 없이 세션을 여는 말을 하면 메뉴를 띄운다. 트리거는
표현이 아니라 **의도**로 판단한다 — "일어나", "시작하자", "진행하자",
"뭐부터 할까", "이어서 하자" 등 대상·범위가 없는 말은 모두 해당. 반대로
구체적 지시("phase 3 이어서 해줘", "이 버그 고쳐줘")로 시작하면 메뉴 없이
바로 수행한다. 여는 말이 다른 스킬·명령의 트리거와 겹쳐 보여도 이 절차가
우선한다 — 실행 스킬은 메뉴에서 액션을 선택받은 뒤에 호출한다.

절차:

0. **서두 멘트를 출력하지 않는다.** "절차대로 문서를 읽을게요" 같은 예고
   문장 없이, 조용히 문서를 읽은 뒤 곧바로 메뉴부터 출력한다.
1. `docs/PROGRESS.md`를 읽는다. 이 가이드와 PROGRESS.md 외의 프로젝트 문서는
   사용자가 선택하기 전에 로드하지 않는다 (아래 3의 phase 설명용 phase
   파일은 예외).
   - Current Phase가 `none`(초기화 전)이면 메뉴를 띄우지 않는다. init-project
     실행(브리프가 비어 있으면 PROJECT_BRIEF.md 작성부터)을 안내한다.
2. **첫 메뉴는 "어디서 작업할지" 선택이다.** 병렬로 존재하는 작업 갈래를
   먼저 보여주고 선택받는다:
   - `docs/areas/`가 존재하면(멀티 모듈/MSA 프로젝트) 영역 메뉴를 띄우고,
     선택받은 뒤 `docs/areas/<영역>.md`를 로드한다.
   - `docs/areas/`가 없어도 `docs/phases/README.md`에 병렬 진행
     가이드(트랙 분리)가 있으면 트랙 메뉴를 띄운다. 각 트랙이 어떤 phase
     들을 다루는지 설명 컬럼에 적고, "전체/공통" 항목(0번)을 포함한다.
   - 둘 다 없으면(단일 트랙) 이 단계는 생략한다.
   - 지난 세션에 작업한 영역/트랙(Working Notes 기록)이 있으면 표시한다.
   영역·트랙은 **컨텍스트 로딩 단위**일 뿐이다 — 진행선(Current
   Phase/Status)은 프로젝트에 하나이며, 어느 영역을 건드리는지는 phase
   파일의 `변경 범위`에 명시한다. 영역별로 독립된 진행선이 필요한 규모라면
   리포 분리를 권장한다.
3. **현재 단계를 설명한 뒤 액션 메뉴를 띄운다.** 트랙/영역이 정해지면
   (또는 그 단계가 생략됐으면) 현재 phase 파일을 읽고, 이 단계가 무엇을
   만드는지(목표)와 작업 목록을 쉬운 말로 요약해 보여준 다음 Status에 맞는
   액션 메뉴를 띄운다. 설명 없이 메뉴만 띄우지 않는다. 번호 대신 자유
   텍스트로 지시하면 그것을 그대로 수행한다 — 메뉴는 라우터지 강제가
   아니다.
4. 선택된 액션에 필요한 문서만 그때 로드한다.
5. **액션이 끝나면 항상 메뉴로 돌아온다.** 결과를 보고한 뒤 갱신된 Status에
   맞는 액션 메뉴를 다시 띄운다. 골든 루프의 "정지"는 다음 phase를 이어가지
   않는다는 뜻이지, 메뉴 없이 침묵한다는 뜻이 아니다 — 세션 내내 메뉴가
   인터페이스다. 메뉴를 생략하는 경우는 세션 마무리 절차 뒤뿐이다.

**메뉴 문구·형식 규칙**:

- 내부 용어(골든 루프, 스킬 이름, Status 마커 원문 등)를 메뉴에 쓰지 않는다.
- 메뉴는 `│ 번호 │ 작업 │ 설명 │` ASCII 표로 표시한다.
- 라벨은 2~4단어 명사형("이어서 개발"), 상세는 설명 컬럼에 쓴다.
- 상태는 표 위에 자연스러운 문장 한 줄로 알린다
  ("Phase 03 — 레시피 검색 작업이 진행 중입니다."). 이전 세션 기록이 있으면
  한 줄 덧붙인다.
- 추천 선택지는 설명 컬럼 끝에 `⭐ 추천`.
- Blockers가 `none`이 아니면 상태 문장 아래 줄에 반드시 표시한다.
- 메뉴 마지막 줄은 항상 "번호를 선택해주세요."

### 영역/트랙 메뉴 (docs/areas/ 또는 병렬 트랙이 있을 때)

```
어느 영역에서 작업하시겠습니까?

│ 번호 │ 영역       │ 설명                       │
│------│-----------│----------------------------│
│  1   │ order-api │ 주문 처리 코어 (지난 세션)  │
│  2   │ admin-web │ 관리자 화면                │
│  3   │ batch     │ 정산 배치                  │
│  0   │ 전체/공통  │ 영역 무관 작업             │

번호를 선택해주세요.
```

트랙 메뉴도 같은 형식으로, 트랙 이름과 담당 phase 흐름을 적는다:

```
어느 트랙에서 작업하시겠습니까?

│ 번호 │ 트랙              │ 설명                                  │
│------│------------------│---------------------------------------│
│  1   │ 트랙 A (개발)     │ Phase 01 뼈대 → 02 이관 → 03 러너 ⭐ 추천│
│  2   │ 트랙 B (지식·라벨)│ Phase 04 위키 → 03 정답 라벨           │
│  0   │ 전체/공통         │ 트랙 무관 작업 (브리핑, 계획 수정 등)   │

번호를 선택해주세요.
```

### 현재 단계 설명 (액션 메뉴 앞에 항상)

액션 메뉴를 띄우기 전에 현재 phase 파일을 읽고 다음을 쉬운 말로 요약한다:

- 이 단계가 끝나면 무엇이 가능해지는지 (목표, 1~2줄)
- 작업 목록 (작업 ID + 한 줄 설명, 완료된 것은 표시)
- 완료 기준 요약 (1줄)

### 액션 메뉴 (Status별)

**NOT STARTED (시작 전):**

```
Phase NN — <이름> 시작 전입니다.

│ 번호 │ 작업        │ 설명                            │
│------│------------│---------------------------------│
│  1   │ 단계 시작   │ 현재 단계 개발 진행 ⭐ 추천      │
│  2   │ 상황 브리핑 │ 진행 상황 정리 (코드 변경 없음)  │
│  3   │ 계획 수정   │ 단계 내용이나 순서 변경          │
│  4   │ 기타 작업   │ 계획에 없는 버그 수정, 실험 등   │

번호를 선택해주세요.
```

**IN PROGRESS (진행 중):**

```
Phase NN — <이름> 작업이 진행 중입니다.
이전 세션: <Working Notes 한 줄>

│ 번호 │ 작업        │ 설명                            │
│------│------------│---------------------------------│
│  1   │ 이어서 개발 │ 하던 단계 마저 완성 ⭐ 추천      │
│  2   │ 상황 브리핑 │ 진행 상황 정리 (코드 변경 없음)  │
│  3   │ 계획 수정   │ 단계 내용이나 순서 변경          │
│  4   │ 기타 작업   │ 계획에 없는 버그 수정, 실험 등   │

번호를 선택해주세요.
```

**AWAITING REVIEW (검토 대기):**

```
Phase NN — <이름> 작업이 끝나 검토 대기 중입니다.

│ 번호 │ 작업             │ 설명                               │
│------│-----------------│------------------------------------│
│  1   │ 테스트 항목 확인 │ 직접 확인할 항목과 실행 명령 ⭐ 추천 │
│  2   │ 수정 반영        │ 검토에서 나온 문제 수정             │
│  3   │ 다음 단계 진행   │ 검토 통과 처리 후 다음 단계 준비     │
│  4   │ 상황 브리핑      │ 진행 상황 정리                      │

번호를 선택해주세요.
```

**DONE (전체 완료):**

```
계획된 모든 단계가 끝났습니다.

│ 번호 │ 작업           │ 설명                       │
│------│---------------│----------------------------│
│  1   │ 다음 계획 수립 │ 새 단계 설계 ⭐ 추천        │
│  2   │ 회고 정리      │ 지금까지의 결정·진행 정리   │
│  3   │ 기타 작업      │ 계획에 없는 작업            │

번호를 선택해주세요.
```

### 각 선택지의 실제 동작

- **단계 시작 / 이어서 개발** — 위 골든 루프를 수행한다.
- **상황 브리핑** — 읽기 전용 상황 보고. 코드·문서를 수정하지 않는다.
- **계획 수정** — 요청에 따라 `docs/phases/` 파일을 수정·추가·삭제하고,
  PROGRESS.md의 Completed 체크리스트를 동기화한다. 코드는 건드리지 않는다.
- **테스트 항목 확인** — PROGRESS.md의 Developer Test 항목을 실행 명령과
  함께 보여준다.
- **수정 반영** — 지적받은 문제를 현재 phase의 `변경 범위` 안에서 고친다.
  시작할 때 Status를 `IN PROGRESS`로 되돌리고, 끝나면 골든 루프 6~7단계
  (UPDATE → COMMIT & STOP)를 다시 수행한다.
- **다음 단계 진행** — Current Phase를 다음 phase로 올리고 Status를
  `NOT STARTED`로, Current Task를 새 phase의 첫 작업으로, Developer Test를
  `none`으로 되돌린다. **마지막 phase였다면** 대신 Status를 `DONE`으로,
  Current Phase와 Current Task를 `none (전체 완료)`으로 정리한다.
- **다음 계획 수립** — 새 기능·마일스톤을 SPEC.md에 반영하고 새 phase 파일을
  추가한 뒤, Completed에 항목을 늘리고 Current Phase를 첫 새 phase로,
  Status를 `NOT STARTED`로 설정한다.
- **회고 정리** — PROGRESS.md·DECISIONS.md·커밋 이력을 바탕으로 회고 요약을
  보고한다. 읽기 전용이며, 사용자가 원할 때만 docs/에 기록한다.
- **기타 작업** — 시작 전에 작업 범위를 먼저 물어 확정하고, 끝나면
  Working Notes에 무엇을 했는지 남긴다.

## 세션 마무리

사용자가 세션을 끝내려는 의도를 보이면 — "오늘 그만하자", "정리하자",
"마무리해줘", "퇴근할게" 등 — 다음을 수행하고 끝낸다:

1. 커밋되지 않은 변경이 있으면 알리고 커밋 여부를 묻는다.
2. `docs/PROGRESS.md`의 Working Notes를 갱신한다: 오늘 한 일 / 남은 일 /
   다음 세션이 알아야 할 함정. (멀티 모듈이면 작업한 영역도 기록)
3. Status가 실제 상태와 맞는지 확인하고, 어긋나면 바로잡는다.
4. 오늘 작업을 한 줄로 요약하고 인사한다.

## 범위 보호

- phase 파일의 `변경 범위` 밖 파일은 수정하지 않는다. 필요하다고 판단되면
  수정하지 말고 Working Notes에 제안으로 적는다.
- SPEC.md의 "제외 범위"에 있는 기능은 어떤 이유로도 구현하지 않는다.
- 스펙에 없는 설계 결정을 내렸다면 `docs/DECISIONS.md`에 한 줄 추가한다.

## 기록 규칙

- 프로젝트 상태는 `docs/PROGRESS.md`가 유일한 소스다. 대화에만 남기지 않는다.
- Status 값은 정확히 다음 네 가지만 사용한다:
  `NOT STARTED` / `IN PROGRESS` / `AWAITING REVIEW` / `DONE`
- Working Notes는 다음 세션의 에이전트가 읽는다고 생각하고 쓴다 —
  무엇을 했고, 무엇이 남았고, 함정이 무엇인지.
- 커밋 메시지는 `docs/CONVENTIONS.md`의 형식을 따른다.

## 토큰 규칙 (컨텍스트 절약)

- `docs/` 전체를 한꺼번에 읽지 않는다. CLAUDE.md의 문서 지도에 따라
  지금 필요한 문서만 읽는다.
- API·이벤트 스펙은 `docs/specs/`의 JSON을 읽는다. 코드를 뒤져 스펙을
  역추적하지 않는다. 스펙을 바꿨으면 JSON도 같이 갱신한다.
- 큰 탐색·조사 작업은 서브에이전트로 분리해 본 컨텍스트를 아낀다.
