# AGENT_GUIDE — 작업 방식

> 이 리포에서 **어떻게** 일하는지. 세션마다 `docs/pipeline.md`와 함께 먼저 읽는다.
> `CLAUDE.md`가 원칙이라면, 이 문서는 절차다. (plick-ai의 AGENT_GUIDE를 이 리포에
> 맞게 재구성 — FastAPI 대신 임베딩 실험·CLI.)

## 1. 골든 루프 (태스크 단위)

진행·태스크는 **한 파일**(`docs/pipeline.md`)에서 통합 관리한다. 현황표(상태) +
페이즈별 태스크 체크박스가 모두 그 파일 안에 있다.

```
1) docs/pipeline.md를 열어 현황표에서 ▶ current 페이즈를 확인한다
2) 그 페이즈 섹션에서 다음 미완료 태스크(- [ ] T#)를 고른다
3) (페이즈의 첫 태스크면) Jira 티켓을 확인·판단(→ §3.1)하고 develop에서
   페이즈/티켓 브랜치를 딴다  (예: feat/KAN-###-openai-matrix)
4) 그 태스크 하나만 구현한다            ← 4원칙 (특히 단순함·외과적)
5) 태스크 뒤 '✔ 검증'을 테스트로 확인한다 (초록)
6) 같은 파일에서 그 태스크를 - [x] 로 체크한다
7) 그 태스크를 페이즈 브랜치에 한 커밋으로 **자동 커밋**한다(승인 불필요)  ← §4
8) 다음 미완료 태스크로 반복. 페이즈의 모든 태스크가 [x]면:
   - 현황표에서 페이즈 상태 ✅ done, 태스크 x/y 갱신, 다음 페이즈 ▶ current
9) 멈춘다. **푸시·develop PR은 사용자 승인 후**(보통 페이즈 끝에 한 번).
```

**단위는 태스크.** 한 번에 태스크 하나씩, 검증까지 초록일 때만 체크·커밋.
**한 태스크 = 한 커밋**. **페이즈 완료 = 그 페이즈의 모든 태스크 [x]**. 막히면
체크·커밋하지 말고 `docs/pipeline.md` 하단 로그에 무엇이 막혔는지 남긴다.

## 2. 소유권 · 접점

- 기본은 단일 개발자. 단 **Phase 05는 2인 병렬 트랙**(트랙 A=Gemini, 트랙 B=OpenAI) —
  트랙별로 독립 설계·실험하고, 공통 기준(데이터셋·지표·노트 양식)만 맞춘다.
  공용 코드(`providers/` 인터페이스·러너·`report/`·`eval/`)를 바꿔야 하면 상대
  트랙과 조율 후.
- **최종 이식처는 plick-ai** — 여기서 고른 최적 구성·파이프라인이 그쪽 AI 서버로
  넘어간다. 구조·컨벤션·Jira(KAN)·Confluence(스페이스 P)는 plick-ai와 공유한다.

## 3. 브랜치 전략

```
main        보호. 릴리스만. 직접 커밋 금지.
 └ develop  통합 브랜치. 페이즈/티켓 브랜치가 여기로 머지된다.
     ├ feat/KAN-###-openai-matrix
     ├ feat/KAN-###-...
     └ chore/<slug>   (티켓 없는 리포 메타 작업)
```

- 페이즈/티켓 브랜치 이름: `<type>/KAN-###-<slug>` (예: `feat/KAN-201-openai-embed`).
  `KAN-###`는 Jira 이슈 키 — 브랜치를 딸 때 실제 티켓 번호로 채운다.
- 흐름: `develop`에서 브랜치 → 한 페이즈 작업·테스트 → 커밋 → 푸시 →
  **develop으로 PR/머지**. 머지 전 `develop`을 최신화(rebase/merge).

### 3.1 브랜치 딸 때 Jira 티켓 확인 (에이전트 필수)
`KAN-###`는 **자리표시자가 아니라 실재하는 Jira 티켓 키**다. 절대 지어내지 않는다.
페이즈/티켓 브랜치를 따기 전 에이전트는 다음을 **직접 확인·판단**한다.

1. **티켓 특정**: `docs/pipeline.md`에서 이 페이즈의 티켓 열·브랜치 표기·로그를 본다.
   번호가 채워져 있으면 그 키.
2. **Jira 대조(MCP)**: 번호가 `###` 자리표시자거나 확신이 없으면 Jira MCP로 확인 —
   `searchJiraIssuesUsingJql`(예: `project = KAN AND summary ~ "<키워드>"`)로 후보를
   찾고, `getJiraIssue`로 **summary·status·issuetype**이 이 태스크와 맞는지 판단.
   임베딩·AI 가공 작업은 대개 **에픽 KAN-3 "[기능] AI 가공"** 하위 작업(Task) 티켓.
3. **type 판단**: 티켓 유형·작업 성격으로 `type`(`feat`/`fix`/`refactor`/`chore`/
   `docs`/`test`)을 고른다.
4. **막히면 멈춘다**: 매칭 티켓이 없거나 후보가 여럿이라 모호하면 **임의로 고르지
   말고 사용자에게 확인**한다.
5. 확정한 키를 브랜치명과 그 브랜치의 모든 커밋 끝 `(KAN-###)`에 **일관되게** 쓴다.
6. **예외 — 리포 메타 작업은 티켓 없이**: 저장소 운영·규칙 문서·툴링 설정은 대응
   티켓이 없으면 KAN 키 없이 `<type>/<slug>` 브랜치 + `(KAN-###)` 없는 `chore:`/
   `docs:` 커밋. 억지 티켓 생성 금지. **진행 문서(pipeline.md)·페이즈 산출물은 메타
   아님** → 항상 페이즈 티켓. 애매하면 사용자에게 확인.

### 3.2 Jira 티켓 생성 규칙
새 작업 티켓을 만들 때(Jira MCP `createJiraIssue`) 아래를 따른다.

- **위치**: 해당 기능 **에픽 아래 작업(Task)** 으로 생성한다. 임베딩·AI 가공 작업은
  에픽 **KAN-3 "[기능] AI 가공"** 하위(`projectKey=KAN`, `issueTypeName=작업`,
  `parent=KAN-3`). 처음엔 **백로그(해야 할 일)** 로 두고, 실제 착수할 때 브랜치를
  따며 상태를 전환한다.
- **제목**: `[AI] - <제목>` 형식. **간결하게(3~5단어)** 쓰되 **제목만 보고 무슨
  일인지 바로 이해**되게. 번역체·전문용어 축약 금지(예: "입력 텍스트 축 확장" ✗ →
  "상세요약 임베딩 비교" ✓).
- **설명**: 쉬운 말로(→ CONVENTIONS "용어"). 양식은 `## 작업 내용`(무엇을·왜) /
  `## 완료 조건`(체크박스) / `## 참고`(상위 Epic·결과물 경로). 기존 KAN-62/63/64와
  같은 골격.
- **생성 전 확인**: 티켓 초안(제목·설명·유형)을 **먼저 사용자에게 보여주고 승인**
  받은 뒤 생성한다. 임의 생성 금지.

## 4. 커밋 규칙

- **커밋은 자동, 푸시는 승인.** 태스크 검증이 초록이면 그 태스크를 자동 커밋한다
  (승인 불필요). **푸시·develop PR은 사용자가 명시적으로 요청할 때만**. `main`/
  `develop` 직접 커밋 금지 — 커밋은 페이즈 브랜치 위에서만.
- 형식: `<type>: <요약> (KAN-###)` — 예 `feat: OpenAI 임베딩 연동 (KAN-201)`.
  `KAN-###`는 §3.1에서 확인한 **실재 티켓 키**(자리표시자 금지).
- **한 커밋 = 한 태스크(T#)**. 한 페이즈의 여러 태스크를 한 커밋에 몰지 않는다.
- **예외 — 리포 메타 작업은 `(KAN-###)` 생략 가능**(§3.1의 6). 프로세스 문서·툴링
  설정 등은 티켓 없이 `chore:`/`docs:`로.
- 요약은 **한글·명령형·50자 이내·마침표 없이**. 본문(선택)은 빈 줄 뒤 '무엇을·왜'만.
  **공동작성자(Co-Authored-By) 트레일러는 넣지 않는다**(plick-ai 컨벤션과 동일).

## 5. 테스트 규칙

- **단위(pytest) = 외부 API 모킹/캐시, 결정론**: 실제 임베딩 API를 호출하지 않는다.
  네트워크 없는 환경에서도 전체 테스트가 초록이어야 한다. 검증 대상 — 군집화·시간
  범위 나누기, 캐시 라운드트립, 채점기(ARI·묶음 정확도·잘못 합침/잘못 나뉨) 결정성,
  위키 링크·목차 정합, 공급자가 캐시만으로 API 없이 도는지.
- **유료 실험은 별도**: 실제 임베딩 API 호출(비용 발생)은 기본 테스트에 넣지 않는다.
  실험 러너로만 돌리고, **돌리기 전 구성·비용을 사용자에게 확인**.
- 각 페이즈의 **[✔ 검증]이 곧 테스트 명세**다. 초록이어야 태스크 체크.

## 5.1 DoD (Definition of Done) — develop 머지 게이트
페이즈 브랜치를 develop에 머지하기 전 아래를 모두 만족:
```
□ 해당 페이즈의 모든 태스크 [x]
□ ruff check + ruff format --check 통과
□ pytest 초록 (외부 API 없이)
□ 실험을 돌렸으면 results/ + wiki/experiments/ 노트 기록
□ 커밋 컨벤션 준수 + pipeline.md 현황표·로그 갱신
```

## 6. 진행 상황 기록
- **태스크 단위**: `docs/pipeline.md`에서 `- [ ] T#` → `- [x] T#`로 체크(검증 초록 후).
- **페이즈 완료 시**: 현황표에서 상태 `✅ done`, 태스크 `x/y` 갱신, 다음 페이즈
  `▶ current`, 막히면 `⛔ blocked: <이유>`.
- 하단 로그에 한 줄 요약을 남겨 다음 세션이 이어받게 한다.
- 스펙 밖 설계 결정은 `docs/DECISIONS.md`에 한 줄 추가한다.

## 7. 실험 진행·분석 방식

실험은 한 번 돌려 최적 구성을 확정하고 끝이 아니라 **반복 루프**로 최고 성능을
노린다: 실험→채점→**잘못 합침/나뉨 사례로 원인 규명**→개선 시도→결과를 `results/`·
`wiki/`에 쌓고 wiki 갱신→다시 반복. (운영 시 임계값 판단은 LLM에 맡기더라도,
벤치마크에선 임계 스윕으로 상한을 보되 **오류 유형을 줄이는 구조적 개선**에 집중.)

- **선정을 서두르지 않는다**: 한 실험 묶음 점수가 나왔다고 곧장 "최고 구성 선정"
  으로 넘어가지 않는다. 먼저 분석하고 남은 축(입력 구성·차원·임계·비교 범위 등)을
  더 돌려 근거가 쌓인 뒤 선정한다.
- **분석엔 사례를 붙인다**: ARI·F1 숫자만 말하지 말고 **구체 사례**를 함께 —
  잘못 합침(어떤 다른 이슈가 뭉쳤나), 잘못 나뉨(어떤 이슈가 쪼개졌나), 그리고
  **잘 묶인 대표 묶음**도. 데이터는 `results/<타임스탬프>/scores.json`(잘못 합침/
  나뉨) + `result.json`(묶음 구성) + 정답 라벨에서 뽑는다.
- 규명된 **실패 유형·개선은 wiki 개념 노트에 누적**한다(예: `잘못 합침과 사가 분할`).
- **임베딩 관련 작업은 매번 "종합 보고서"를 갱신한다** — 시도 / 어떤 문제가 어느
  리포트에서 나왔나 / 무엇을 개선했나 / before→after 수치 / 최초 대비 총 개선폭까지.
  형식은 `wiki/트랙B_OpenAI_실험종합.md`를 본보기로. **두 트랙 모두** 각자 자기 보고서를
  같은 형식으로 유지한다(트랙 A=`wiki/트랙A_Gemini_실험종합.md`, 트랙 B=위 파일).
  **두 트랙이 끝나면 Phase 06에서 둘을 나란히 비교하는 보고서**(모델·task_type별 수치,
  강·약점, 최종 선정 근거)를 같은 형식으로 만든다.
- **보고서는 발표·인수인계에 그대로 쓸 수 있어야 한다** — 처음 보는 사람도 읽히게
  쉬운 말로. 전문 용어·번역체를 새로 만들지 말고(→ `docs/CONVENTIONS.md` "용어"),
  방법은 "실제로 무엇을 했는지" 명확히 적고("합치는 조건을 엄격하게 = 가장 안 비슷한
  기사끼리도 기준을 넘어야 합치도록" 처럼), 필요한 기술어(ARI 등)는 첫머리에 한 줄로
  풀어 준다. 소제목은 대화체 말고 간결하게. "처음 보는 사람이 이해할까?"를 점검한다.
