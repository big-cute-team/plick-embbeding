# SPEC

## 한 줄 정의

PLick의 "같은 이슈 묶기(원문 N : 요약 1)"를 위한 임베딩 모델 벤치마크·실험
리포 — 최적 임베딩 구성을 정량 실험으로 선정하고, 그 구성으로
수집→임베딩→벡터 저장→증분 중복 묶기 파이프라인을 검증한다. 검증된 결과는
plick-ai(FastAPI)로 이식한다.

## 핵심 기능

1. **PoC 이관·재현** — plick-ai/poc-embedding 파이프라인(임베딩 → 24h 롤링
   윈도우 → 병합형 군집화(Agglomerative) → 리포트)을 이 리포에서 재현 가능하게 이관.
2. **정량 평가 기반** — 실데이터에 정답 라벨을 붙이고 ARI(군집 일치 점수)·쌍 단위(pairwise)
   정밀도·재현율 러너 구축 (현재는 눈 검증뿐).
3. **LLM Wiki (Obsidian)** — 리포 내 `wiki/` 보관함(vault). 실험 1회마다 결과
   노트가 자동 생성되어 쌓이고, 모델·개념 노트로 지식을 누적해 두 사람이
   서로의 실험을 참고한다.
4. **모델별 병렬 실험 + 선정** — 2인이 모델을 하나씩 맡아(Gemini /
   OpenAI text-embedding-3) task_type × 차원 × 임계 실험을 병렬 진행,
   같은 지표로 채점해 wiki에 기록 → 종합 비교로 최적 구성 선정.
5. **증분 중복 묶기** — 발행 시각 순 처리(직전 24h 활성 이슈와 비교),
   기존 실험에서 드러난 예외 케이스(같은 맥락·다른 선수 오병합, 여러 날 이어지는 이슈가
   24h 윈도우에 쪼개지는 사가 분할) 검증 + 회색지대 LLM 검수 실험.
6. **수집→임베딩→저장 파이프라인** — 선정 구성으로 기사 로드 → 임베딩 →
   벡터 저장 → 증분 중복 묶기까지 배치 실행 가능한 파이프라인.

## 제외 범위

- **HTTP 서버(FastAPI 등)** — 실험·배치 리포다. 서비스화는 plick-ai 합류로
  결정하며, 이 리포에서 서버를 띄우지 않는다.
- **LLM 요약 재생성** — 티어 기반 요약 갱신 결정 트리 실행은 plick-ai 영역.
  이 리포는 "같은 이슈인지 판단"까지만 다룬다.
- **개인화 추천** — 자문 3 영역, 별도 트랙.
- **운영 MySQL 스키마 쓰기** — 원본 데이터는 읽기 전용. 실험 산출물은
  로컬(results/)·전용 저장소에만 쓴다.
- **자체 임베딩 모델 학습·파인튜닝** — API 기반(Gemini·OpenAI) 비교만.
- **어드민 UI** — 없음.

## 용어

- **이슈** — 같은 사건을 다룬 원문 묶음. 피드 1개의 단위 (원문 N : 요약 1).
- **오병합** — 다른 이슈(예: 같은 팀·다른 선수)가 한 군집으로 묶이는 오류.
- **회색지대** — 임계 부근이라 임베딩만으로 판단이 애매해 LLM 검수가 필요한 구간.
- **task_type** — 임베딩 API의 용도 파라미터(SEMANTIC_SIMILARITY, CLUSTERING 등).
  같은 텍스트라도 벡터·코사인 스케일이 달라진다.
- **24h 롤링 윈도우** — 새 글을 발행시각 T 기준 [T−24h, T]에 갱신된 활성
  이슈하고만 비교. >24h 잠잠 후 재등장 = 새 이슈.
- **ARI (군집 일치 점수)** — Adjusted Rand Index. 정답 군집 대비 제안 군집의 일치도 지표.
- **사가 분할** — 여러 날 이어지는 이슈(사가)가 24h 윈도우 때문에 여러
  이슈로 갈리는 현상.
