# Phase 08 — 수집→임베딩→벡터 저장 파이프라인

## 목표

선정된 구성으로 실제 기사 데이터를 로드해 임베딩하고, 벡터를 저장하고,
증분 중복 묶기까지 이어지는 배치 파이프라인이 반복 실행 가능하다. plick-ai
이식 기준(무엇을 어디로 옮기는지)이 문서화된다.

## 작업

- [x] P08-T01 벡터 저장 방법 조사·결정 — 벡터를 어디에 저장할지 비교한다:
      로컬 파일 또는 MySQL(JSON·BLOB 컬럼 + model·task_type·dim·normalized
      메타 정보) vs 벡터 전용 DB(Chroma 등). 지금 규모(하루치 후보를 전부
      비교해도 되는 수준)를 기준으로 결정하고 DECISIONS에 기록한다
      → **결정**: 이 리포는 로컬 파일 저장소로 검증, 운영은 Postgres(메타)+전용
      벡터DB(벡터) 분리, 벡터DB는 plick-ai 이식 때 연결 (DECISIONS 2026-07-23, KAN-290)
- [ ] P08-T02 기사 불러오기 — 원본 저장소(읽기 전용)에서 새로 들어온
      기사만 가져오는 증분 로더를 만든다 (접속 정보는 개발자가 제공)
- [ ] P08-T03 전체 배치 파이프라인 — 불러오기 → 임베딩(캐시 활용) → 벡터
      저장 → 증분 중복 묶기 → 기사-이슈 매핑 저장까지 한 번에 이어지게
      한다. 다시 실행하면 이미 처리한 기사는 건너뛰고 이어서 처리한다
- [ ] P08-T04 실패 대비 — API 오류가 나면 재시도하고, 중간에 끊겼다가
      다시 돌려도 이미 처리한 기사를 중복 처리하지 않는지 검증한다
- [ ] P08-T05 plick-ai 이식 가이드 쓰기 — 확정된 구성, 어떤 모듈을 어떤
      경계로 옮길지, 인터페이스 계약 초안을 `docs/`에 정리한다
      (실제 이식 작업은 plick-ai 리포에서 한다)
