# Phase 06 — 결과 종합 · 최적 구성 선정

## 목표

두 트랙의 wiki 기록을 종합해 우리 서비스의 임베딩 구성(모델 · task_type ·
차원 · 임계)을 근거와 함께 확정한다. 이 결정이 Phase 07~08과 plick-ai
이식의 기준이 된다.

## 작업

- [ ] P06-T01 종합 비교표 만들기 — 두 트랙의 실험 노트에서 점수를 모아
      (모델 × 구성)별로 ARI·쌍 단위 F1·임계값 유지 구간 폭·비용/쿼터를
      한눈에 볼 수 있는 비교표를 만든다 (노트 머리말(frontmatter)을
      스크립트로 집계)
- [ ] P06-T02 점수 외 조건 평가 — 최고 점수만 보지 않고 함께 평가한다:
      임계값이 조금 틀어져도 안정적인지(유지 구간 폭), API 비용·호출
      한도(쿼터)·응답 속도, 우리 팀 기술 스택과 잘 맞는지
- [ ] P06-T03 빈 곳 메우기 — 비교표에서 아직 안 해본 조합(공백)이 보이면
      해당 조합을 추가로 실험한다 (양 트랙 협의)
- [ ] P06-T04 최종 결정 — 어떤 구성을 쓸지와 그 근거를 wiki 결정 노트와
      `docs/DECISIONS.md`에 기록하고, 팀 공유용(Confluence 마스터 문서 §11)
      요약을 작성한다

## 변경 범위

`wiki/`, `scripts/`, `docs/PROGRESS.md`, `docs/DECISIONS.md`

## 완료 조건

- [ ] 두 트랙 전 실험이 포함된 종합 비교표가 wiki에 있다
- [ ] 선정 구성이 점수·임계 구간 폭·비용 3개 축의 근거와 함께
      `docs/DECISIONS.md`에 기록돼 있다
- [ ] 탈락 구성에 대해서도 "왜 아닌지" 한 줄씩 기록돼 있다
- [ ] 팀 공유용 요약(Confluence 게시 가능한 형태)이 준비돼 있다

## 개발자 테스트

- [ ] 비교표를 두 사람이 함께 검토하고 선정에 합의
- [ ] 선정 구성으로 대표 실험 1회 재실행 — 기록된 점수가 재현되는지 확인
