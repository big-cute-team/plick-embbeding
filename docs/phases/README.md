# phases/

프로젝트를 단계(phase)로 쪼갠 작업 명세. 파일명은 `NN-이름.md`
(`01-project-skeleton.md`처럼 0을 채운 두 자리 번호). 한 phase는
한 세션 안에 끝낼 수 있는 크기로 잡는다 — 크면 쪼갠다.

에이전트는 한 번에 정확히 한 phase만 수행하고 정지한다 (골든 루프,
`docs/AGENT_GUIDE.md` 참조).

## phase 파일 템플릿

아래 섹션 제목은 **글자 그대로** 유지할 것 — 워크플로우 스킬들이
이 제목을 기준으로 동작한다.

```markdown
# Phase NN — <이름>

## 목표

<이 phase가 끝나면 무엇이 가능해지는가, 1~3줄>

## 작업

- [ ] PNN-T01 <작업 한 줄>
- [ ] PNN-T02 <작업 한 줄>

## 변경 범위

<수정해도 되는 파일/디렉토리. 이 밖은 제안만 하고 건드리지 않는다.>

## 완료 조건

<객관적으로 점검 가능한 항목만. "잘 동작한다" 금지, "X 하면 Y가 된다" 형식.>

- [ ] <조건 1>
- [ ] <조건 2>

## 개발자 테스트

<phase 완료 후 개발자가 직접 확인할 것. 명령어/URL 포함, 복붙 가능하게.>

- [ ] <확인 항목 1>
```

## 병렬 진행 가이드 (Phase 01~04, 2인)

Phase 01~04는 의존성이 갈라져 있어 두 사람이 트랙을 나눠 병렬로 진행할 수
있다. "한 세션 = 한 phase" 원칙은 유지된다 — 사람(세션)이 다르면 서로 다른
phase를 동시에 진행해도 된다. 두 트랙의 변경 범위(`src/`·`scripts/` vs
`wiki/`·`data/labels/`)가 겹치지 않아 충돌이 없다. 공유 파일은
`docs/PROGRESS.md`뿐이므로 갱신 시 git pull 먼저.

**트랙 A (개발)** — Phase 01 → Phase 02 → Phase 03의 지표 러너

**트랙 B (지식·라벨)** — Phase 04의 보관함·템플릿·지식노트·백필(즉시 시작
가능) → 데이터 스냅샷이 나오면 Phase 03의 정답 라벨

티켓 분리 (Jira 발행용 6장):

| # | 담당 | 내용 | 선행 조건 |
|---|------|------|-----------|
| 1 | A | Phase 01 전체 — 프로젝트 뼈대 | 없음 |
| 2 | A | Phase 02 전체 — PoC 이관·재현 (**P02-T01 데이터 스냅샷을 최우선으로** — 티켓 4를 풀어줌) | 티켓 1 |
| 3 | B | P04-T01·T02·T04·T05 — 보관함 구조·노트 템플릿·지식 노트·기존 실험 백필 | 없음 (즉시 시작) |
| 4 | B | P03-T01·T02 — 라벨 포맷 정의·published 56건 정답 라벨 부여 | P02-T01 (데이터 스냅샷) |
| 5 | A | P03-T03·T04·T05 — 지표 러너·리포트 통합·기준 점수 측정 | 티켓 2, 4 |
| 6 | A/B | P04-T03 — 실험 실행 시 wiki 노트 자동 생성 연동 | 티켓 2, 3 |

합류 지점: 티켓 5·6이 끝나면 Phase 05(모델별 실험, 원래 2인 병렬 설계)로
자연스럽게 이어진다.

## 작성 규칙

- **완료 조건**은 에이전트가 자가 점검하는 기준이다. 측정 불가능한 조건은 쓰지 않는다.
- **변경 범위**는 하드 펜스다. 좁을수록 안전하다.
- **개발자 테스트**는 phase 완료 시 PROGRESS.md의 Developer Test 섹션으로 복사된다.
- 작업 ID는 `PNN-TNN` 형식 (예: `P03-T01`) — 진행 기록에서 이 ID로 참조한다.
- **작업 항목은 풀어 쓴 쉬운 문장으로 쓴다.** 키워드 축약체("X 모듈 구현 —
  A·B·C 파라미터") 대신 무엇을 왜 만드는지 읽히는 문장으로 쓰고, 필요한
  전문용어는 괄호로 병기한다. 세션 시작 메뉴에서 이 문장이 그대로
  사용자에게 보인다.
