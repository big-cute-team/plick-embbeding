## §6 시스템 아키텍처에 추가 — 임베딩·중복 묶기의 역할 경계 (2026-07-14 확정)

**별도 임베딩 서버는 두지 않는다.** 의미적 중복(원문 N : 요약 1 묶기)은
기능2 v2로서 **plick-ai(FastAPI) 안에 모듈로 편입**한다. 근거:

- 처리 흐름이 분류 → 요약 → 임베딩 → 후보 비교 → 회색지대 LLM 판정 →
  (병합 시) 재요약으로 LLM 호출과 얽혀 있어, 서버를 나누면 요청 한 건에
  서버 간 왕복·프롬프트 중복·계약 2개가 생긴다.
- 임베딩은 모델 서빙이 아니라 API 호출(gemini-embedding-001)이고, 수집량
  (기자 10명 × 일 10~15건)·비교 후보(24h 수십 건) 규모가 분리를 정당화하지
  않는다. 부하가 커지면 답은 서버 분리가 아니라 같은 코드베이스 +
  브로커/워커(멘토링 문서 §12.4).

**역할 경계 (쓰기 일원화 원칙의 확장):**

| 무엇 | 누가 | 비고 |
| --- | --- | --- |
| 임베딩 생성·유사도 비교·회색지대 LLM 판정·병합 재요약 | plick-ai | 읽기 전용 연산 — 무상태(stateless) 유지. 비교 후보군(24h 활성 이슈의 대표 임베딩·팀태그)과 기자 티어는 어드민이 요청에 동봉 |
| MySQL 쓰기 (원문 연결·요약 생성/갱신·태그·status) | admin-server | 기존 원칙 그대로 — 한 트랜잭션 |
| 임베딩(벡터) 저장 | **어드민 MySQL 커밋 이후** 트리거 | 아래 참조 |

**벡터 저장은 반드시 커밋 후.** 새 이슈의 벡터 키(`article_summary_id`)는
어드민이 INSERT해야 생기고, 커밋 전에 벡터를 쓰면 어드민 트랜잭션 실패 시
"MySQL엔 없는 이슈의 벡터"(유령 벡터)가 남아 이후 기사가 존재하지 않는
이슈에 병합되는 사고가 난다. 따라서 검수·발행의 Slack 알림과 같은
after-commit 자리에서 벡터를 저장한다(어드민 직접 저장 또는 plick-ai의
저장 엔드포인트 호출). 저장 실패는 fail-safe(해당 기사가 다음 비교에 안
잡힐 뿐) — 백필로 복구.

**저장소 선택**: 당분간 벡터DB 없이 MySQL 컬럼(JSON/BLOB + model ·
task_type · dim · normalized 메타) — 중복 처리 파이프라인 문서의 기존 결정
유지. 벡터DB 승격 여부는 실험 리포 Phase 08에서 재검토.

## §5.3 / 기능2 v2에 추가 — 같은 이슈 판정 규칙 3겹 (2026-07-14 확정)

top-1 매핑(가장 유사한 기사에 무조건 연결)은 금지 — 가장 유사한 기사는
항상 존재하므로 새 이슈가 어딘가에 강제 병합된다. 판정은 세 겹:

```
① 카테고리 분리    — 같은 카테고리(TRANSFER/MATCH/FITNESS)끼리만 비교.
                     임계값은 카테고리별로 따로 튜닝.
                     (OTHER·AI 장애 fallback 글은 예외적으로 전체 비교)
② 임계 + 회색지대  — 임계 이상 → 병합 후보 / 회색지대 → LLM에 "같은
                     이슈인가" 판정 / 미만 → 새 이슈
③ 인물 가드        — 임계를 넘어도 공유 인물(figure)이 없으면 자동 병합
                     금지, 회색지대로 강등해 LLM 회부
```

근거(실험 기록 폴더 2026-06-29, published 56건): 임계 0.85에서 "같은
팀·같은 맥락의 **다른 선수**"가 병합되는 잘못 합침 2건(제임스+라이스,
기마랑이스+토날리) 실측. 두 건 모두 같은 카테고리 안에서 발생 →
카테고리 분리만으로는 못 잡고 ③ 인물 가드가 필요. 인물 정보는 v1의
사전 매칭(figures/figure_aliases)이 이미 산출하므로 추가 비용 0.

## 중복 처리 파이프라인 문서(15171586) 갱신분 — 병합 후 요약 갱신 규칙 개정

**기존**: 새 기자 티어 < 대표 → "함께 보도"만 추가, 요약 유지.
**개정(2026-07-14)**: 낮은 티어의 글에 **새 정보(디테일)가 있으면 버리지
않고 서술 강도를 낮춰 편입**한다. 중복 묶기의 목적이 정보 풍부화이므로
"왼쪽 무릎, 전치 4주" 같은 실질 정보를 버리는 건 손해.

- **보완 vs 충돌 구분**: 높은 티어 내용에 디테일을 더하는 것(보완)은
  헤지 표현으로 편입 — "호날두가 부상으로 출전이 어렵다. 일부 매체에서는
  왼쪽 무릎 전치 4주라는 이야기도 전해진다." / 높은 티어와 모순(충돌)이면
  편입하지 않고 높은 티어 서술 유지.
- **티어 → 서술 강도 매핑**: 티어 1~2 = 단정 서술("~로 출전이 어렵다"),
  티어 3~4 = 전문(傳聞) 서술("~라는 보도도 있다"). 프롬프트에 원문들과
  각 기자 티어를 함께 주고 이 매핑을 지시.
- **근거 인용 강제**: 요약의 각 주장이 어느 원문에서 왔는지 인용하게 해
  여러 기사 접합 시의 환각 방어(v1 팀태그 방식과 동일).

**재임베딩 규칙(기존 미결정 항목 해소)**: 요약 텍스트가 바뀌는 모든
분기(대표 교체 재요약·같은 기자 최신 교체·동티어 재생성)에서 대표 임베딩을
재계산한다 — 트리거는 위의 after-commit 저장 지점과 동일. 임베딩 대상
텍스트는 "제목 + 짧은 요약"으로 고정해, "함께 보도" 한 줄 추가 같은
잔변경에는 벡터가 흔들리지 않게 한다(재임베딩 불필요 + 비교 안정성).

## §9.3 기능2 / §11에 추가 — 임베딩 실험 리포(plick-embbeding) 신설

임베딩·중복 묶기 실험 전용 리포를 신설했다(2026-07-14). **프로덕션 코드는
여기서 만들지 않는다** — 실험·검증·결정만 하고, 확정된 구성과 규칙을
plick-ai에 이식한다.

**Phase 구조(8단계)**: 01 뼈대 → 02 PoC 이관·재현(published 56건 기준) →
03 정답 라벨 + 정량 지표(ARI) → 04 LLM Wiki(Obsidian) → 05 모델별 실험
(2인 병렬: Gemini / OpenAI) → 06 최적 구성 확정 → 07 증분 중복 묶기 +
예외 케이스(회색지대 LLM·인물 가드·카테고리 분리·사가 분할 대응) →
08 배치 파이프라인 + plick-ai 이식 가이드.

Phase 01~04는 2인 병렬 트랙으로 진행(트랙 A 개발 / 트랙 B 지식·라벨 —
티켓 6장 분리, 리포 `docs/phases/README.md` 참조).

**서버 갱신 시점**:

- Phase 06 완료 = 임베딩 구성(모델·task_type·차원·임계) 확정 — 공유만
- Phase 07 완료 = 판정 규칙 확정 — **plick-ai 중복 묶기 모듈의 스펙**
- Phase 08 완료 = 저장 방식 확정 + 이식 가이드 — **서버 작업 착수 문서**

**이식 시 계약 영향(README §2 조율 필요)**: plick-ai `/classify` 응답에
`embedding` + 중복 판정(`new_issue` 또는 `matched_summary_id`·유사도·판정
방법) 추가 / 어드민 요청에 후보군(24h 활성 이슈 대표 임베딩·팀태그)과
기자 티어 동봉 / 어드민은 판정 소비(원문 연결 + 티어 결정 트리 + 요약
갱신을 한 트랜잭션) 후 커밋 시점에 벡터 저장.

**운영 도입은 섀도 모드부터**: 초기에는 중복 판정을 저장·로깅만 하고 자동
병합은 끈 채(병합 후보는 REVIEW로) 관측 후 활성화 — 멘토의 "새 군집 바로
발행 금지" 원칙과 동일 맥락. 잘못 합침의 사용자 노출을 차단.
