# OpenAI 임베딩 실험 종합 보고서

처음 보는 사람도 읽을 수 있게 쉬운 말로 쓴다. 숫자의 근거는 각 실험 노트
(`experiments/`)와 `results/<날짜시간>/` 폴더에 있고, 날짜순 기록은 `docs/DECISIONS.md`.
데이터는 기사 90건(사람이 정한 정답은 56개 사건).

## 용어

- **같은 이슈 묶기**: 같은 사건을 다룬 기사들을 한 묶음으로 모으는 일. 이 실험의 목표.
- **임베딩**: 기사를 숫자 목록으로 바꿔, 컴퓨터가 기사끼리 얼마나 비슷한지 재게 하는 것.
- **기준값**: 두 기사가 이 값보다 비슷하면 "같은 사건"으로 묶는다. 높을수록 덜(깐깐하게) 묶는다.
- **점수(ARI)**: 우리 묶음이 사람이 만든 정답과 얼마나 똑같은지를 0~1로 잰 값. 1이면 완벽.

## 1. 출발점

- 비교 대상은 구글의 Gemini 모델. 정답과의 일치도(점수) **0.9071**.
- OpenAI 모델을 처음 측정했을 때는 **0.7545** — Gemini보다 0.15 낮게 시작했다.

목표: OpenAI를 여러 방법으로 개선해 Gemini에 얼마나 다가갈 수 있는지 보는 것.

## 2. 발견한 문제

| 문제 | 설명 | 어느 실험에서 |
|------|------|-------------|
| 이름만 같으면 합쳐버림 | 서로 다른 사건인데 같은 선수·팀 이름이 나오면 한 묶음이 됨. 예: 홀란 관련 서로 다른 소식 4개가 한 덩어리 | `results/…144917` |
| 같은 사건이 여러 개로 쪼개짐 | 며칠에 걸쳐 나온 한 사건의 기사들이 하나로 안 묶이고 여러 묶음으로 갈라짐. 예: 알론소의 첼시 감독 부임이 3개 묶음으로 | `results/…150724` |
| 요약이 비어 있는 기사 | 기사 한 건(7351)이 요약 단계에서 "우리 팀과 무관"으로 걸러져, 진짜 요약 대신 "검수 필요" 안내문만 있었다. 내용이 비어 엉뚱하게 떨어짐 | `results/…150724` |
| 언어가 다르면 안 묶임 | 같은 사건이라도 한글 기사들 사이에 영어 기사 하나가 끼면 따로 떨어짐 | `results/…150724` |
| 기준값을 성글게 봄 | 기준값을 0.05 단위로만 확인해서, 그 사이(0.01 단위)에 있던 진짜 최고 점수를 놓쳤다 | 3의 ① |

문제 유형은 [[잘못 합침과 사가 분할|잘못 합치고 쪼개는 문제]], 쓸 수 있는 정보는 [[묶기에 쓸 신호와 구조]].

## 3. 개선 시도와 점수 변화

각 줄은 "실제로 무엇을 했나 → 점수 변화". ✅ 좋아짐 / ⚪ 효과 없음(그래도 기록).

| 한 일 | 실제로 어떻게 했나 | 점수 변화 | 판정 |
|------|------------------|----------|------|
| ① 기준값을 더 잘게 확인 | 같은 사건으로 볼 기준값을 0.05 단위가 아니라 0.01 단위로 시험해 최고 점수 지점을 찾음 | 0.7545 → **0.7976** | ✅ |
| ② 제목만 넣기 | 임베딩에 넣는 글에서 요약을 빼고 제목만 넣음 | 0.7976 → 0.7082 | ⚪ 나빠짐(요약이 도움 됨) |
| ③ 긴 요약 넣기 | 짧은 요약 대신 더 긴 자세한 요약을 넣음 | 0.7568 → **0.8077** | ✅ (차원이 큰 1536에서 특히) |
| ④ 시간 제한 끄기 | "발행이 24시간 안쪽인 기사끼리만 비교" 규칙을 없애고 전부 비교 | 0.8077 → 0.7980 | ⚪ 거의 그대로 |
| ⑤ 비어 있던 요약 채우기 | 요약이 안내문뿐이던 기사(7351)에 제대로 된 요약을 넣음 | 0.8077 → **0.8636** | ✅ 한 건인데 크게 오름 |
| ⑥ 다른 분야면 안 묶기 | 기사 분야(이적·경기·부상 등)가 서로 다르면 아예 안 묶음 | 0.8077 → **0.8126** | ✅ |
| ⑦ 핵심 대상이 다르면 안 묶기 | 기사의 핵심 대상(선수·팀·감독)이 서로 다르면 아예 안 묶음 | 0.8126 → **0.8701** | ✅ 크게 오름 |
| ⑧ 합치는 조건을 엄격하게 | 두 묶음을 합칠지 볼 때, 원래는 '평균으로 비슷하면' 합쳤는데, '가장 안 비슷한 기사끼리도 기준을 넘어야' 합치도록 바꿈 | 0.8077 → 0.7522 | ⚪ 나빠짐 |
| ⑨ 공통 성분 빼기 | 모든 기사가 공통으로 가져서 구분에 도움 안 되는 숫자 방향을 임베딩에서 빼봄 | 0.8077 → 0.80 | ⚪ 안 오름 |

> ⑤(요약 채우기)와 ⑦(핵심 대상)은 같은 7351 문제를 다른 방법으로 푼다 — ⑤는 요약을
> 직접 고쳐서, ⑦은 제목에서 대상(틸레망스)을 뽑아 자동으로 제자리로 돌려놓는다.

## 4. 누적 개선 과정

가장 좋았던 구성 기준으로, 점수가 어떻게 쌓여 왔는지.

| 단계 | 무엇을 했나 | 점수 | 직전보다 |
|------|-----------|-----|---------|
| 출발 | OpenAI 첫 측정 | 0.7545 | — |
| +기준값 잘게 | 기준값을 0.01 단위로 확인 | 0.7976 | +0.043 |
| +긴 요약 | 자세한 요약을 넣음 | 0.8077 | +0.010 |
| +분야·대상 구분 | 분야·핵심 대상이 다르면 안 묶음 | **0.8701** | +0.062 |

## 5. 결론

- **처음 0.7545 → 최고 0.8701. 총 +0.116(약 15%p) 좋아졌다.** Gemini와의 차이도
  0.15에서 **약 0.06**으로 좁혔다(양쪽 다 공정하게 맞춘 기준 Gemini 0.9287).
- **임베딩 자체로 짜낼 건 다 짜냈다.** 모델 크기·차원(숫자 개수)·입력 텍스트·기준값,
  그리고 합치는 조건·공통 성분 제거까지 해봤고, 자세한 요약을 넣은 0.8077이 임베딩만의 한계.
- **여기서 더 올린 건 임베딩이 아니라 "분야·핵심 대상으로 먼저 나눠 묶는" 방식이었다**
  (0.8077 → 0.8701). 남은 애매한 한 건(홀란의 서로 다른 발언)은 사람처럼 뜻을 읽는
  LLM이 맡아야 한다. → 정리한 방향: **분야 → 핵심 대상 → 임베딩 → LLM** ([[묶기에 쓸 신호와 구조]]).
- **읽을 때 주의**: ⑤(요약 고침)·⑦(핵심 대상)은 이번엔 사람이 손으로 만들어 넣은
  가정 실험이다(실제 서비스에선 LLM이 한다). 최종 모델 선택(Gemini vs OpenAI)은
  다음 단계(Phase 06)에서, Gemini 쪽 측정이 끝난 뒤에 한다.
