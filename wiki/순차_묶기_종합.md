# 순차(하나씩) 중복 묶기 종합 보고서 — Phase 07

처음 보는 사람도 읽을 수 있게 쉬운 말로 쓴다. 데이터는 기사 90건(사람이 정한 정답은
56개 사건). 숫자의 근거는 각 실험 노트와 `docs/DECISIONS.md`. 임베딩 구성 선정은
[[최종선정_임베딩구성]](Phase 06)에서 끝났고, 이 문서는 **그 임베딩으로 "실시간에
하나씩 들어오는 기사"를 어떻게 묶을지**를 다룬다.

## 용어

- **배치(한꺼번에)**: 기사를 다 모아 한 번에 비교해 묶는 방식. 지금까지의 실험 방식.
- **순차(하나씩)**: 새 기사가 올 때마다 그때그때 묶는 방식. 실제 서비스의 흐름.
- **대표값**: 한 묶음을 요약하는 벡터 하나. 새 기사를 이것과 비교해 붙일지 정한다.
- **기준값**: 두 기사가 이 값보다 비슷하면 같은 사건으로 본다(0.86).
- **점수(ARI)**: 우리 묶음이 정답과 얼마나 똑같은지 0~1로 잰 값. 1이면 완벽.
- **잘못 합침 / 잘못 나뉨**: 다른 사건을 한 묶음으로 / 같은 사건을 여러 묶음으로.

## 1. 출발점

임베딩(Gemini·SEMANTIC·768·짧은요약·기준값 0.86)으로 **배치**로 묶으면 ARI **0.9287**.
하지만 운영은 기사가 하나씩 들어온다. 목표는 이 하나씩 흐름을 재현하고, 배치에 최대한
가깝게 끌어올리며, 그 과정에서 드러난 예외를 실험으로 정리하는 것.

## 2. 발견한 문제

| 문제 | 설명 | 어디서 |
|------|------|-------|
| 하나씩이 더 합침 | 순차로 바꾸니 배치보다 잘못 합침이 늘었다(1→최대 4) | [[배치_vs_순차_묶기_비교]] |
| 대표값이 넓어짐 | 묶음이 커질수록 평균 대표값이 뭉뚱그려져 다른 사건까지 끌어들임 | [[순차_묶기_대표값_비교]] |
| 애매한 경계 | 유사도 0.80~0.90에 잘못 합침·잘못 나뉨이 몰려 있어 임베딩만으론 못 가름 | [[애매한_구간_LLM_판정]] |
| 여러 날 사가 | 며칠 걸친 사건이 24시간 범위를 넘겨 갈림(에데르송·알론소 등) | [[사가_대응_비교]] |

문제 유형 정리는 [[잘못 합침과 사가 분할]], 쓸 수 있는 신호·구조는 [[묶기에 쓸 신호와 구조]].

## 3. 개선 시도와 점수 변화

각 줄은 "실제로 무엇을 했나 → 점수 변화". ✅ 좋아짐 / ⚪ 효과 없음(그래도 기록).

| 태스크 | 한 일 | 실제로 어떻게 | 점수 변화 | 판정 |
|------|------|------------|----------|------|
| T01 | 하나씩 묶기 엔진 | 발행 시각 순으로 한 건씩, 최근 24h 살아있는 묶음과 비교해 배정 | 순차 방식 마련 | ✅ |
| T02 | 배치 vs 순차 비교 | 같은 데이터로 두 방식 채점 | 배치 0.9287 vs 순차(평균) **0.8415** | — 격차 확인 |
| T03 | 대표값 3안 | 평균 / 최신 / 첫 기사를 각각 채점 | 0.8415 → **0.8636**(첫 기사) | ✅ 첫 기사 채택 |
| T04 | 애매구간 LLM | 0.80~0.90만 OpenAI에 "같은 이슈?"를 묻되 **묶음 전체**를 보여줌 | 0.8636 → **0.9149**, 잘못 합침 0 | ✅ 큰 개선 |
| T05 | 사가 대응 | 범위 늘리기 vs 수명 연장 채점 | 둘 다 무효(늘리면 오히려 하락) | ⚪ 24h 유지 |

**최초(순차 평균) 0.8415 → 최종 0.9149**, 총 **+0.073**. 배치(0.9287)와의 남은 격차는
0.014뿐이고, 그 대부분은 T05의 사가 분할(잘못 나뉨 7건)이다.

### T04가 결정적이었던 이유

처음엔 새 기사를 묶음의 **첫 기사 하나**와만 비교해 물었더니, LLM이 이적 사가의 단계
(협상 vs 메디컬)를 다른 사건으로 보고 쪼개 **ARI가 0.6925로 붕괴**했다. **묶음 전체(지금까지의
스토리)를 보여주고 "이슈=이어지는 한 사건"으로 정의**하니, 사가는 지키면서 잘못 합침만
0으로 없앴다. LLM 호출은 애매 구간 39건뿐(묶음 하나 최대 ~600토큰, 전체 ~1센트).

## 4. 최종 순차 구성

```
임베딩:   Gemini gemini-embedding-001 · SEMANTIC_SIMILARITY · 768차원 · 제목+짧은요약
묶기:     발행 시각 순 하나씩 · 기준값 0.86 · 최근 24시간 · 대표값=첫 기사(seed)
검수:     유사도 0.80~0.90만 OpenAI(gpt-4o-mini)로 "같은 이슈?" 판정(묶음 전체 제시)
결과:     ARI 0.9149 · 잘못 합침 0 · 잘못 나뉨 7(사가)
```

- 임베딩(Gemini)·판정(OpenAI) 응답은 캐시 → 재실행 비용 0.
- 테스트 54개(외부 API 없이) 초록.

## 5. 남은 과제 · plick-ai 이식

- **남은 잘못 나뉨 7건 = 여러 날 사가.** 시간 규칙으론 이득 없이 못 푼다(늘리면 잘못 합침↑).
  진짜 해법은 시간이 아니라 **주체 연결** — "에데르송/Ederson/맨유 골키퍼"를 날짜 건너 한
  인물로 이어 주기. 이건 추출·표준화 문제라 plick-ai의 주체 추출(KAN-275)이 담당한다.
- **이식 형태**: 실시간 파이프라인은 기사 1건 도착 → 요약·주체·카테고리(1단계 LLM) →
  임베딩 1회 저장 → (카테고리·주체·최근 N시간) 필터로 최근접 조회 → 기준 넘으면 붙이고
  애매하면 LLM 판정. 벤치마크의 이 순차 로직·기준값·대표값·검수 방식이 그대로 넘어간다.
- 배치 파이프라인(수집→저장) 검증은 Phase 08.
